Java ClassLoader Delegation Model

Every java developer must bear this in mind. This is the default java class loading mechanism.

Let’s illustrate the following snippet of code:

import sun.net.spi.nameservice.dns.DNSNameServiceDescriptor;

public class DefaultLoading {
    public static void main(String[] args) {
        System.out.println("DefaultLoading loader:" + DefaultLoading.class.getClassLoader());
        String m = "hello";
        System.out.println("String loader:"+ m.getClass().getClassLoader());
        System.out.println("DNSNameServiceDescriptor loader" + DNSNameServiceDescriptor.class.getClassLoader());
        System.out.println("Duck loader:" + Duck.class.getClassLoader());
    }
}

When I run java -cp . DefaultLoading in a directory that contains DefaultLoading.class, on my machine this outputs:

DefaultLoading loader:sun.misc.Launcher$AppClassLoader@73d16e93
String loader:null
DNSNameServiceDescriptor loadersun.misc.Launcher$ExtClassLoader@33909752
Duck loader:sun.misc.Launcher$AppClassLoader@73d16e93

Here’s what happened:

  1. By default, each loader will delegate to its parent to when a class is seen and must be resolved. This occurs recursively until we reach the bootstrap loader. If and only if none of its upstream ancestry loader could load the class will the loader try to load it itself. In this case, When JVM finds the entry class DefaultLoading is not loaded yet, it tries to load with system loader,which in turn delegates to extension loader. Finally the extension loader delegates to the bootstrap loader. However, DefaultLoading class is not part of the java core library, nor is it placed in extension loader’s path: jre/lib/ext/, so it was loaded by system loader(sun.misc.Launcher$AppClassLoader@73d16e93).
  2. Since String class is part is part of java.lang package, it’s loaded by the bootstrap loader, which is null in the output.
    Actually, the same is true for System class and PrintStream class(which is the static out field of System).
    Note that classes loaded by any of the upstream ancestor is visible to the current loader, but NOT vice versa.
  3. DNSNameServiceDescriptor is in jre/lib/ext/dnsns.jar. That’s why its loader is ExtClassLoader;
  4. As the out field in System, Duck will be loaded by the loader that loaded DefaultLoading. It could be loaded by neither bootstrap nor extension loader, for it’s only available on your class path. All delegation loading fails, it’s loaded finally by the same loader of DefaultLoading, i.e. AppClassLoader@73d16e93

Merge Two Sorted Linked List

Steps to merge List1 and List2:

  1. Create one temporary Node as the head of the target list. Two pointers H, L points to the head;
  2. Loop until List1 or List2 becomes null, in each iteration make the node with smaller value the next node of target list,
    breaks the original chain.
  3. If List1 is null, chain List2 to target list, otherwise append List1 to target list;
  4. Remove the temporary head, return the succeeding node.

Loop invariant: Target list is sorted, pointer L points to last node, node(s) in target list break from original list.

  1. Before the while loop, apparently the assertion is true;
  2. After loop one, it still holds as illustrated in the diagram;
  3. When the while loop exits, the lower list is empty, L stays at its last node, the upper list stands apart from the sorted one.

Swap Two LinkedList Node without Exchanging Value

First prepare some testing data to better understand this problem, or to unveil some pitfalls before jumping to code.

  • Some node might not exist in the linked list
  • Node might be the head or the tail
  • Node might be wellformed, with predecessor and successor
  • Node might be adjacent
  • Predecessor and successor might overlap
  • Node might be null

Formulate the General Procedure:

  1. Loop once to save pointers to the two nodes to be swapped
  2. Suppose node1 precedes node2 in LinkedList,
    predecessor of node1 precedes node2
    successor of node1 succeeds node2
    predecessor of node2 precedes node1
    successor of node2 succeeds node1

However, this doesn’t work if the two nodes are next to each other. Just need to reverse the direction in this case.

Regex Atomic Group Demystified

“An atomic group is a group that, when the regex engine exits from it, automatically throws away all backtracking positions remembered by any tokens inside the group. Atomic groups are non-capturing.”

For a better comprehension, let’s experiment with some examples.

Match (?>\d+)2 against “42”, RegexBuddy fails after two attempts.

Watch closely how it fails at first attempt. Quantifier + is greedy by default, eating as much digits as possible, leaving no chance for 2 to match for no characters are left in the testing string. Then it reports failure immediately for the attempt at first character WITHOUT BACKTRACKING.

This is exactly how possessive quantifier works in Java, it’s equivalent to \d++2. Now you should be able to figure out why
(?>at|ato)m fails to match “atom” but ((?>at)|ato)m succeeds.

So the key point of atomic group is to suppress backtracking behavior while not keeping the matching group.
